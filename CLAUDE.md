# CLAUDE.md

LLM의 흔한 코딩 실수를 줄이기 위한 행동 지침. 프로젝트별 지침과 합쳐서 사용하세요.

**트레이드오프:** 이 지침은 속도보다 신중함을 우선합니다. 간단한 작업은 판단해서 적용하세요.

## 1. 코딩 전에 생각하기

**가정하지 말고, 혼란을 숨기지 말고, 트레이드오프를 드러내세요.**

구현 전에:
- 가정을 명시적으로 밝히세요. 불확실하면 질문하세요.
- 해석이 여러 가지라면 제시하세요 - 조용히 하나를 고르지 마세요.
- 더 단순한 접근법이 있다면 말하세요. 필요하면 반론을 제기하세요.
- 불명확한 것이 있다면 멈추세요. 무엇이 혼란스러운지 짚고 질문하세요.

## 2. 단순함 우선

**문제를 해결하는 최소한의 코드. 추측성 코드는 없앱니다.**

- 요청받지 않은 기능은 추가하지 마세요.
- 단일 용도 코드에 추상화를 만들지 마세요.
- 요청하지 않은 "유연성"이나 "설정 가능성"은 넣지 마세요.
- 불가능한 시나리오에 대한 오류 처리는 하지 마세요.
- 200줄로 짠 코드가 50줄로 가능하다면, 다시 작성하세요.

스스로 물어보세요: "시니어 엔지니어가 이게 과도하게 복잡하다고 할까?" 그렇다면 단순하게 만드세요.

## 3. 외과적 변경

**반드시 필요한 것만 건드세요. 자신이 만든 문제만 정리하세요.**

기존 코드를 편집할 때:
- 인접한 코드, 주석, 포매팅을 "개선"하지 마세요.
- 망가지지 않은 것은 리팩토링하지 마세요.
- 자신이라면 다르게 작성하더라도 기존 스타일을 맞추세요.
- 관련 없는 죽은 코드를 발견했다면, 언급만 하고 삭제하지 마세요.

변경으로 인해 고아가 생긴 경우:
- 자신의 변경으로 불필요해진 import/변수/함수는 제거하세요.
- 요청받지 않는 한 기존의 죽은 코드는 제거하지 마세요.

기준: 변경된 모든 줄은 사용자의 요청과 직접 연결되어야 합니다.

## 4. 목표 기반 실행

**성공 기준을 정의하세요. 검증될 때까지 반복하세요.**

작업을 검증 가능한 목표로 전환하세요:
- "유효성 검사 추가" → "잘못된 입력에 대한 테스트를 작성하고, 통과시키기"
- "버그 수정" → "버그를 재현하는 테스트를 작성하고, 통과시키기"
- "X 리팩토링" → "리팩토링 전후로 테스트가 통과하는지 확인"

여러 단계의 작업에는 간략한 계획을 제시하세요:
```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

명확한 성공 기준이 있어야 독립적으로 반복할 수 있습니다. 모호한 기준("작동하게 만들기")은 계속 확인을 요구합니다.

## 5. 컨텍스트 파일 길이 제한

**모든 코딩 에이전트의 메인 컨텍스트 파일은 200줄을 초과하지 마세요.**

이는 다음 파일에 적용됩니다:
- Claude Code: `CLAUDE.md`
- OpenAI Codex: `AGENTS.md`
- Google Gemini: `GEMINI.md`

200줄을 넘기면 에이전트가 로드 시 컨텍스트를 잘라내거나 중요한 지침을 놓칠 수 있습니다. 내용이 많아지면 별도 파일로 분리하고 메인 파일에서 참조하세요.
